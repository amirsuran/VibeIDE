# Лучшие практики написания скиллов

Детальное руководство по авторингу эффективных скиллов в `.vibe/skills/`.

## Progressive disclosure (послойная загрузка)

Скиллы загружаются послойно ради экономии контекста:

1. **Метаданные (name + description)** — загружены всегда.
2. **Тело SKILL.md** — загружается при срабатывании скилла.
3. **Файлы references/** — загружаются только при необходимости.

### Когда выносить в references

SKILL.md держать в пределах 150–200 строк. При перерастании:

**Паттерн 1 — гайд верхнего уровня со ссылками.** SKILL.md содержит ядро процесса и указывает на детали:

```markdown
## Продвинутое

- **Детальная конфигурация:** [references/config.md](references/config.md)
- **Справочник API:** [references/api.md](references/api.md)
- **Примеры:** [references/examples.md](references/examples.md)
```

**Паттерн 2 — организация по доменам.** Для скиллов с несколькими независимыми областями:

```
skill-name/
├── SKILL.md (обзор и навигация)
└── references/
    ├── domain-a.md
    ├── domain-b.md
    └── domain-c.md
```

Работа идёт с domain-a → агент грузит только domain-a.md.

**Паттерн 3 — условные детали.** Базовое инлайн, продвинутое по ссылке:

```markdown
## Базовое использование

[ядро инструкций]

**Продвинутая настройка:** [references/advanced.md](references/advanced.md)
```

### Важные правила

- References — на один уровень: все файлы ссылаются напрямую из SKILL.md.
- Никаких вложенных references (ссылок из reference на другой reference).
- Reference длиннее 100 строк — оглавление в начале.

## Описания (description)

По description агент решает, загружать ли скилл. Это главный механизм обнаружения.

1. **Конкретика и ключевые термины.** Хорошо: «Добавление feature flag для изоляции изменений». Плохо: «Помогает с фичами».
2. **Что + когда.** Что делает («Написание и прогон unit-тестов») + когда применять («в этой кодовой базе»).
3. **Третье лицо.** Хорошо: «Добавляет feature flags». Плохо: «Я помогу добавить…», «Ты можешь использовать это…».
4. **Триггер-термины.** Конкретные файлы, команды, концепции: «Использовать при работе с PDF, формами, извлечением из документов». Английские термины (`merge`, `PRD`, `rebase`) оставлять — пользователь пишет ими.

## Принципы лаконичности

Окно контекста делится между всеми скиллами, историей диалога и системным промптом. Каждый токен на счету.

### Базовое допущение: агент уже умный

Добавлять только то, чего агент не знает. К каждому куску вопрос: «агенту правда нужно это объяснение?», «оправдывает ли абзац свою токен-цену?»

**Хорошо (лаконично):**

```markdown
## Извлечение текста из PDF

Использовать pdfplumber:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

**Плохо (многословно):** «PDF (Portable Document Format) — распространённый формат файлов, содержащий текст, изображения… Существует много библиотек, но мы рекомендуем pdfplumber, потому что… Сначала установите через pip…» — агент знает, что такое PDF и как работают библиотеки.

## Оформление примеров кода

- **Всегда указывать язык** блока кода (```python, ```bash, …).
- Для workflow-скиллов — «до/после» или пошагово.
- Shell-команды — полностью, с флагами: `npm test -- --watch --coverage`. Неочевидные флаги пояснять, но предпочитать самодокументируемые команды.

## Workflow против простых инструкций

Нумерованные шаги — для многошаговых процессов, где важен порядок. Для сложных workflow добавлять копируемый чек-лист прогресса:

```
Прогресс:
- [ ] Шаг 1: анализ формы
- [ ] Шаг 2: маппинг полей
- [ ] Шаг 3: валидация
- [ ] Шаг 4: заполнение
- [ ] Шаг 5: проверка
```

Для прямолинейных задач структуру workflow пропускать.

## Антипаттерны

### ❌ Windows-пути
Только прямые слэши: `scripts/helper.py`, не `scripts\helper.py`.

### ❌ Размытые описания
«Извлечение текста и таблиц из PDF», а не «Помогает с документами».

### ❌ Слишком много вариантов
Не перечислять подходы без необходимости. Плохо: «можно pypdf, или pdfplumber, или PyMuPDF, или…». Хорошо: «Использовать pdfplumber; для сканов с OCR — pdf2image + pytesseract».

### ❌ Привязка ко времени
Без дат и версионных оговорок («до августа 2025 используйте старый API»). Вместо этого — секции «Текущий способ» и «Устаревшие паттерны» с пометками о депрекации.

### ❌ Разнобой терминологии
Один термин на всё: всегда «API endpoint», а не вперемешку «URL», «route», «path».

### ❌ Объяснение очевидного
Не «Git — это система контроля версий, которая…», а «`git --no-pager diff` — diff без пагинации».

### ❌ Пере-структурирование простых скиллов
Не каждому скиллу нужны Overview, Best Practices и Examples. Простой — заголовок + инструкции; средний — плюс Суть; сложный — полная структура.

## Нейминг

Рекомендуется глагольная форма действия или предметная пара:

- действие: `resolve-merge-conflicts`, `review-pr`, `write-tech-spec`
- предмет: `pdf-processing`, `roadmap-autopilot`

Избегать: размытых `helper`, `utils`, `tools`; слишком общих `documents`, `data`, `files`.

## Итерация скиллов

Скиллы улучшаются через использование:

1. **Наблюдать** — где агент буксует, где справляется.
2. **Найти пробел** — какой информации не хватило, что было неясно.
3. **Точечно обновить** — починить конкретное, не пере-объясняя всё.
4. **Проверить** — прогнать скилл на похожей задаче.

Итерации держать сфокусированными: не добавлять контент впрок — только то, чья необходимость доказана реальным использованием. После содержательного изменения — поднять `vibeVersion`.
