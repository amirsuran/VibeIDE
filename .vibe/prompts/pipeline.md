# pipeline.md — маршрут проекта до продакшена

<!-- В чате: /my:pipeline -->
<!-- Живой чек-лист готовности. Агент считает этот файл источником истины и не даёт проскочить блокирующие этапы (🔒). -->

Веди работу по маршруту ниже как по **единственному источнику истины о готовности проекта к выкату**.

**Протокол агента** (работает в любой модели и инструменте — Claude Code, Claude.ai, Cursor):

1. Нет файла `.vibe/pipeline.md` — создай его из этого маршрута и скажи об этом.
2. В начале сессии назови **текущий этап** (первый сверху не-`[x]`) и работай внутри него.
3. Отмечай прогресс прямо в файле: `[ ]` → `[~]` (в работе) → `[x]` (закрыт). Это единственное место учёта.
4. Иди строго сверху вниз. Этапы 🔒 — **блокирующие**: релиз без них запрещён.
5. Просят выкатить, а любой 🔒 выше не `[x]` — **откажись** и ответь ровно: «Стоп. По pipeline.md не закрыты блокирующие этапы: <список>». Предложи закрыть их первыми.
6. Не ставь `[x]` без выполненного критерия. «Работает у меня» — не критерий; критерий — **воспроизводимая проверка** (команда, тест, событие в дашборде, скриншот). Перед `[x]` одной строкой напиши, чем проверено.
7. Этап неприменим (нет денег / нет LLM / нет внешних API) — пометь весь этап `[x] N/A: <причина>`. Неприменимость — тоже решение, его фиксируем.
8. Не выдумывай. Не знаешь — проверь код/доки или спроси. «Наверное» и «должно работать» на 🔒-этапах запрещены; непроверенное помечай «НЕ ПРОВЕРЕНО».
9. Правишь маршрут под свой проект — уважаю твою версию, но блокирующие этапы не удаляются, только помечаются N/A с причиной.

---

## 1. Идея и гипотеза                     [ ]
- [ ] Одна фраза: **кто** пользователь и **какую боль** закрываем
- [ ] Чем это лучше того, как он решает боль сейчас (в т.ч. «никак» — тоже способ)
- [ ] Метрика успеха: число и срок (например, «10 платящих за 30 дней»)
- [ ] Антискоуп: 3–5 вещей, которых в MVP **не будет** — записать, чтобы не расползлось

## 2. Архитектура · BaaS-first             [ ]
- [ ] Выбран BaaS (Supabase/Firebase/Convex/…), а не свой сервер без реальной нужды; свой — записать, почему BaaS не подошёл
- [ ] Схема данных: таблицы, связи, владелец каждой строки — продумано ДО кода
- [ ] Модель доступа: роли (anon/user/admin) и что каждой можно — таблицей
- [ ] Оценена стоимость инфры на 0 / 100 / 10 000 пользователей (на старте — около нуля)
- [ ] Хостинг фронта, домен, платёжный провайдер (если деньги) выбраны и совместимы с юрисдикцией

## 3. Секреты и конфигурация          🔒   [ ]
- [ ] Все ключи/токены/пароли — только в env (или секрет-менеджере), **ни одного в коде**
- [ ] `.env` в `.gitignore`; в репо есть `.env.example` с пустыми значениями
- [ ] История git проверена на утечки (поиск `key`, `secret`, `token`, `sk-`, `password`); найденное **отозвано и перевыпущено**, а не просто удалено из кода
- [ ] Админ-ключи (service-role, приватные) живут только на сервере, не попадают в клиентский бандл
- [ ] Ключи prod и dev разные; dev не имеет доступа к боевым данным
- [ ] Секреты не пишутся в логи и тексты ошибок

## 4. Аутентификация и сессии              [ ]
- [ ] Auth через готовый провайдер (Supabase/Firebase Auth, Clerk, OAuth), **не самописная** крипта
- [ ] Восстановление пароля и подтверждение email работают вживую (письма доходят)
- [ ] Rate-limit на вход/регистрацию/восстановление — брутфорс и перебор email невозможны
- [ ] Сессии: выход работает, токен истекает; роль назначается на сервере — клиент не сделает себя админом
- [ ] Регистрация ботов затруднена (captcha / подтверждение email / лимит на IP)

## 5. Данные и доступ · RLS            🔒   [ ]
- [ ] RLS (или эквивалент) включён на **каждой** таблице; нет таблицы «включу потом»
- [ ] Для каждой таблицы записано: кто читает / пишет / удаляет
- [ ] Проверено вживую: анонимный и другой авторизованный не читают и не меняют чужие строки (2 запроса «от чужого лица» — оба отклонены)
- [ ] Критичные операции (баланс, платежи, роли, лимиты) идут только на сервере / в защищённых функциях
- [ ] Хранилище/бакеты: приватны по умолчанию, доступ по правилам, а не по «секретной» ссылке
- [ ] Каскадные удаления продуманы: удаление пользователя не ломает чужие данные и не плодит сирот

## 6. Интеграции · доки-first              [ ]
- [ ] Каждый внешний API сверен с **официальной докой актуальной версии**, а не «по памяти» модели
- [ ] Ошибки интеграций обработаны: timeout, 429, 5xx — приложение деградирует мягко, не падает
- [ ] Вебхуки: подпись проверяется, повторная доставка не портит данные (идемпотентность)
- [ ] Лимиты и цена каждого API известны — что будет при 100× трафика, понятно заранее
- [ ] Ключи интеграций — по этапу 3 (env, не в клиенте)

## 7. AI-слой · если есть LLM              [ ]
*(Нет LLM-фич — пометь «N/A».)*
- [ ] Ключи провайдера LLM только на сервере; клиент ходит через твой backend, не напрямую
- [ ] Лимиты на пользователя: запросы/токены в час — бесплатный юзер не сожжёт бюджет
- [ ] Стоимость действия оценена в токенах, известен месячный бюджет на 100 активных
- [ ] Prompt injection: ввод пользователя не перезаписывает системные инструкции (проверено попыткой)
- [ ] Вывод модели валидируется перед использованием (структура/типы); сырой вывод не исполняется и не пишется в БД
- [ ] Фолбэк при недоступности провайдера: очередь/ретрай/честное сообщение, а не белый экран

## 8. MVP и база UX                        [ ]
- [ ] Основной happy-path работает end-to-end и запускается локально одной командой
- [ ] Ошибки и пустые состояния: у каждого экрана есть «загрузка / пусто / ошибка», не только «успех»
- [ ] Мобильная вёрстка: основной сценарий проходится с телефона (проверено на ширине ≤390px)
- [ ] Формы: валидация с человеческими сообщениями; повторный сабмит не создаёт дубли
- [ ] Без спекулятивных абстракций — минимум кода, решающий задачу

## 9. Сквозные e2e-тесты               🔒   [ ]
- [ ] Ключевые сценарии покрыты e2e: вход/регистрация, основной ценностный поток, оплата (если есть деньги)
- [ ] «Несчастливые» пути: отклонённая оплата, невалидный ввод, потеря сети, повторный сабмит
- [ ] Тесты не зависят друг от друга и от «вчерашних» данных; каждый готовит своё состояние
- [ ] Зелёные локально И на чистом окружении / в CI, запуск одной командой
- [ ] Правило «на красном не деплоим» записано

## 10. Безопасность · атака на прод    🔒   [ ]
- [ ] IDOR: подстановка чужих id в каждый эндпоинт/URL — чужие данные не читаются и не меняются
- [ ] Инъекции: SQL/NoSQL/command на каждом вводе; XSS на всём, что рендерит пользовательский ввод
- [ ] Нет открытых бакетов, debug-эндпоинтов, `/admin` без авторизации, утечки стеков в ошибках
- [ ] Подмена цены/суммы/роли с клиента отклоняется сервером; скрытые поля и disabled-кнопки — не защита
- [ ] Гонки: параллельные запросы на списание/промо/лимит не дают двойного эффекта
- [ ] Rate-limit на всём чувствительном; заголовки безопасности и CORS не `*` на приватных API
- [ ] Зависимости: `npm audit` (или аналог) без critical; lock-файл в репо
- [ ] Атаки прогнаны **по боевому окружению** (или его точной копии), не только локально
- [ ] Каждая находка: закрыта или записана с планом и сроком

## 11. Производительность                  [ ]
- [ ] Нет N+1 и неиндексированных запросов на горячем пути (по EXPLAIN/логам, не «на глаз»)
- [ ] Проверено на близком к реальному объёме: сотни/тысячи строк, а не 3 тестовые
- [ ] Пагинация везде, где список может расти; лимиты по умолчанию
- [ ] Тяжёлое (отчёты, экспорт, LLM) — асинхронно или с прогрессом; UI не блокируется
- [ ] Вес фронта под контролем: бандл без лишних мегабайт, картинки сжаты

## 12. Продуктовая аналитика           🔒   [ ]
- [ ] Каждый шаг онбординга — отдельное событие: открыл → зарегистрировался → сделал ключевое действие → дошёл до ценности
- [ ] Ключевые бизнес-события: активация, покупка, депозит, вывод, отмена
- [ ] Воронка показывает, сколько дошло и отвалилось на каждом шаге
- [ ] События названы по схеме, а не «event1/test/final2»
- [ ] События **реально приходят в дашборд** — проверено вживую на себе, а не «код написан»

## 13. Наблюдаемость и бэкапы              [ ]
- [ ] Ошибки собираются (Sentry/аналог) с алертом — о падении узнаёшь ты, а не пользователь
- [ ] Мониторинг доступности основного URL с уведомлением (telegram/email)
- [ ] Логи ключевых операций есть и читаемы при разборе инцидента
- [ ] Бэкапы БД включены И **проверены восстановлением** (бэкап без теста восстановления — лотерея)
- [ ] Понятно, сколько данных теряется при аварии (день? час?) — и это осознанный выбор

## 14. Страховки на деньги             🔒   [ ]
*(Денег в продукте нет — «N/A». Есть — этот этап обязателен.)*
- [ ] Суммы платежей/депозитов проверяются на сервере; ничего с фронта не берётся на веру
- [ ] Платёжные вебхуки: подпись проверена, повторы идемпотентны, «успех» — только от провайдера, не от клиента
- [ ] Ручная модерация депозитов/выводов на старте (или лимиты + подтверждение крупных)
- [ ] Сумму можно поправить/откатить до зачисления; подозрительный аккаунт — заморозить
- [ ] Алерты на аномалии: резкий рост баланса, серия выводов, всплеск по промо
- [ ] Журнал денежных операций, по которому потом восстановим инцидент

## 15. Правовое и приватность              [ ]
- [ ] Условия использования и политика приватности опубликованы и доступны с каждой страницы
- [ ] Указан продавец (реквизиты по юрисдикции) и способ связи
- [ ] Персональные данные: собираем минимум; что и зачем — в политике
- [ ] Согласия, где нужны (рассылка, cookie-трекинг — по юрисдикции аудитории)
- [ ] Условия возврата указаны (особенно для цифровых товаров) и совпадают с настройками провайдера
- [ ] Support-email существует и читается

## 16. Релиз в прод                    🔒   [ ]
- [ ] Prod и dev разделены: окружения, ключи, БД; деплой в прод — осознанное действие, не «случайно запушил в main»
- [ ] Миграции обратимы; план отката записан и **отрепетирован один раз**
- [ ] Прогон по этапам 1–15: все 🔒 закрыты — не «почти»
- [ ] Задеплоено + живой smoke с чужого устройства/сети: открывается, регистрируется, платит
- [ ] Аналитика ловит реальные события первых пользователей; мониторинг и алерты включены

---

*Маршрут завершён, когда продукт в проде: защищён, покрыт e2e и аналитикой, юридически оформлен и — если есть деньги — со страховками. Дистрибуция и маркетинг намеренно вне области: маршрут заканчивается на «в проде и готов принимать трафик».*
