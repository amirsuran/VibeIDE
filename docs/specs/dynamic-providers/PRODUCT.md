# PRODUCT — Динамические LLM-провайдеры как полноценные built-in (объявление через файл)

## Summary

Провайдер, объявленный в `.vibe/providers.json`, должен быть **неотличим от встроенного** во всём, кроме способа объявления (файл вместо кода): он появляется в Настройках по `order`, ему вводится ключ, при наличии ключа его модели **подтягиваются из `<baseURL>/v1/models`** (как у встроенных, не из файла), пользователь включает нужные — и только они появляются в выборе модели чата. Транспорт и reasoning (уже реализованы) делают так, что выбранная модель реально отвечает и «думает».

## Problem

Сейчас динамический провайдер: (1) требует **вшитого** списка моделей в файле; (2) **не виден** в Настройках (ни карточкой, ни в «Моделях»), ключ ввести негде; (3) его модели попадают в дропдроп **без ключа** и **без включения** — в отличие от встроенных, где модель появляется только у настроенного провайдера после включения. Это непоследовательно и не соответствует ожидаемому потоку.

## Goals

- Динамик ведёт себя как built-in: Настройки → ключ → авто-каталог моделей → включение → дропдроп.
- Модели **не хардкодятся** в файле — тянутся из `/v1/models` провайдера.
- Модель в дропдропе только у **подключённого (с ключом)** провайдера и только если она **включена**.
- Нулевая регрессия встроенных.

## Non-goals

- Поддержка провайдеров **без** OpenAI-совместимого `/v1/models` — для них остаётся опциональный `models.static` (см. инвариант 6). Авто-каталог целевой, static — запасной.
- Перенос FIM/автокомплита на динамики — отдельно.
- Произвольная интерполяция/скрипты в файле — формат остаётся декларативным.

## Behavior

### Объявление через файл

1. В `.vibe/providers.json` провайдер описывается **только как подключение**: `id`, `name`, `active`, `order`, `protocol`, `baseURL`, `auth`, источник ключа (`apiKeyEnv`/`apiKeyRef`), заголовки, таймаут. **Секция `models` не обязательна** — по умолчанию модели берутся из каталога провайдера (инвариант 5).

2. Совпадение `id` со встроенным — патч встроенного; новый `id` — новый провайдер; `extends: "<id>"` — клон. Это поведение сохраняется.

### Видимость и подключение в Настройках

3. Динамический активный провайдер **отображается в Настройках → «Облачные провайдеры»** отдельной карточкой (как встроенные), **в порядке `order`** относительно других ВАШИХ провайдеров. В карточке есть **поле ввода ключа** и (если заданы) ссылки `docsUrl`/`apiKeyUrl`. Ввод ключа здесь сохраняет его в защищённое хранилище VibeIDE (в `.vibe/providers.json` ключ не пишется никогда).

4. **Ключ — «подключение».** Провайдер считается подключённым, если ключ резолвится (поле ключа в Настройках → защищённое хранилище, ЛИБО `apiKeyEnv` из `.vibe/.env`, ЛИБО `apiKeyRef`). Источник, видимый рендереру (Настройки/`.vibe/.env`/`apiKeyRef`), нужен для UI-гейтинга; ключ только из OS-переменной рендереру не виден → в UI провайдер будет считаться «без ключа» (см. инвариант 12).

### Каталог моделей (не из файла)

5. У **подключённого** провайдера модели **подтягиваются автоматически** из `<baseURL>/v1/models` тем же механизмом «каталог моделей», что у встроенных рефрешабельных провайдеров (в Настройках есть действие **«Обновить каталог моделей <провайдер>»** и для динамика). Без ключа каталог не запрашивается.

6. `models.static` в файле — **опционально**: эти модели добавляются/пиннятся поверх каталога (для провайдеров без `/v1/models` или чтобы зафиксировать/донастроить конкретные id). Их caps (`contextWindow`, `toolFormat`, `reasoning`, `extraBody`, …) применяются к совпадающим по `id` моделям каталога. Файл без `models` — нормальный кейс.

### Включение и появление в дропдропе

7. Во вкладке **Настройки → «Модели»** динамический провайдер показан группой (как встроенные) с его моделями (из каталога + static); каждую можно **включить/выключить**.

8. В **дропдропе выбора модели в чате** появляются **только включённые** модели **подключённых** провайдеров. Динамики идут **по `order`, выше встроенных**; модели без включения или у провайдера без ключа — **не показываются**.

9. Выбранная динамическая модель **реально отвечает и думает** (существующие транспорт + reasoning): запрос уходит на её `baseURL`, reasoning/`extraBody` применяются. (Этот слой уже реализован — здесь он должен продолжать работать.)

### Состояния и края

10. **Нет ключа** → провайдер виден в Настройках карточкой с подсказкой «введите ключ»; каталог не тянется; в дропдропе его моделей нет. Это не ошибка.

11. **Каталог недоступен** (оффлайн/5xx/нет `/v1/models`) → понятная диагностика у действия «Обновить каталог»; если есть `models.static` — показываются они; иначе у провайдера 0 моделей (не падение).

12. **Ключ только в OS-переменной** (не в Настройках и не в `.vibe/.env`) → рендерер его не видит, UI считает провайдер «без ключа» (моделей в дропдропе нет). Чтобы подключить через UI — ввести ключ в поле Настроек или положить в `.vibe/.env`. (Документируется явно, чтобы не выглядело багом.)

13. **Без `baseURL`** (и не унаследован) → провайдер не маршрутизируется и не тянет каталог; карточка показывает причину.

14. **Изменения подхватываются:** правка `.vibe/providers.json` и значение ключа в `.vibe/.env` — на лету (как сейчас); структурные изменения, требующие рестарта, явно обозначены.

### Не-регрессия

15. Поведение, карточки, каталоги и форма запроса **встроенных** провайдеров не меняются. При отсутствии `.vibe/providers.json` всё идентично состоянию до фичи.

## Open questions

- **✅ Решено (каталог):** ключ есть → авто-каталог по умолчанию (проба `<baseURL>/v1/models`; нестандартный путь — строкой в `models.fetch`); `models.fetch: false` — отключить авто-каталог, использовать только `static`. Реализовано в `vibeDynamicProvidersService._enrichWithCatalog`.
- **Open question (хранение ключа):** ключ динамика — в общий защищённый стор по id или в отдельный — решится в TECH (затрагивает persisted-секреты).
