# PRODUCT — Динамические LLM-провайдеры: реальная отправка + reasoning + надёжность под Moonshot/Kimi

## Summary

Провайдеры и модели, заданные пользователем в `.vibe/providers.json` (без пересборки IDE), должны не только появляться в выборе модели, но и **реально работать в чате**: модель отвечает, при необходимости «думает» (reasoning) и корректно вызывает инструменты. Цель — довести динамические провайдеры от «видны в списке» до «полноценно работают, как встроенные», на примере Kimi/Moonshot (`api.moonshot.ai`), не сломав ни один встроенный провайдер.

## Problem

Сейчас динамический провайдер появляется в дропдауне моделей, но запрос **не уходит**: маршрутизация отправки распознаёт только встроенные имена провайдеров, а для пользовательского id завершается ошибкой «провайдер не распознан». Даже когда отправка заработает, модели Kimi/Moonshot без дополнительной настройки не переходят в режим размышления и в части случаев отклоняют схемы инструментов или возвращают пустой ответ. Транспортная заготовка (ключ/baseURL/заголовки) уже собирается, но до отправляющего кода не доходит.

## Goals

- Динамический провайдер с заданным транспортом реально отправляет запросы и отвечает в чате.
- Динамическая модель «думает» (reasoning) и надёжно вызывает инструменты, когда провайдер это поддерживает.
- **Нулевая регрессия** встроенных провайдеров.
- Секрет (ключ) никогда не хранится в `.vibe/providers.json`.

## Non-goals

- Авто-список моделей (`fetch: true`) для динамических провайдеров — отдельный follow-up; здесь модели задаются статически.
- UI-поле ввода ключа для динамических провайдеров — отдельная фича.
- Слияние baseURL встроенного для `extends`-провайдера без явного `baseURL` — follow-up (см. инвариант 5).
- FIM/автокомплит для динамических провайдеров — позже.

## Behavior

### Отправка и маршрутизация

1. Провайдер, заданный в `.vibe/providers.json` с `active: true`, статической моделью и резолвимым транспортом (есть `baseURL` и доступный ключ), **отправляет реальный запрос**: выбранная динамическая модель отвечает в чате потоково (стриминг), как встроенная — без пересборки IDE.

2. **Резолв ключа — фиксированный приоритет:** `apiKeyRef` (ключ из защищённых настроек) → значение из `.vibe/.env` по имени `apiKeyEnv` → переменная окружения процесса `apiKeyEnv`. Берётся первый непустой источник. Изменение значения в `.vibe/.env` подхватывается **без перезапуска**; ключ из переменной окружения ОС требует перезапуска (читается при старте процесса).

3. **Без `baseURL` — не тихий провал.** Если у динамического провайдера нет резолвимого `baseURL` (и он не унаследован), отправка не происходит молча: пользователь получает понятное сообщение с причиной («не задан baseURL»), а не зависание и не общая ошибка «провайдер не распознан».

4. **Ключ не найден ни в одном источнике** → понятная ошибка с подсказкой, где задать ключ (`apiKeyEnv` в `.vibe/.env` или защищённые настройки), а не молчаливый отказ. (Ответ апстрима 401/403 при неверном ключе транслируется пользователю как ошибка авторизации, не как пустой результат.)

5. **`extends`-провайдер без собственного `baseURL`** в текущей версии не маршрутизируется (известное ограничение, см. Non-goals): он не предлагается как отправляемый, и пользователь видит причину. Это поведение явно зафиксировано, чтобы не выглядело как баг.

6. **Инструменты у динамической модели.** В режимах с инструментами (agent/обычный) динамическая модель получает тот же набор инструментов (встроенные + MCP), что и встроенные провайдеры, вызывает их, и результаты возвращаются в тот же агентный цикл — включая существующую устойчивость (нормализация имён/алиасов, восстановление кривых вызовов, текстовый XML-fallback для моделей без нативного tool-calling).

7. **Авто-подбор протокола.** Если динамическая модель по факту использует иной wire-протокол (например, модель на агрегаторе, которая на самом деле Anthropic-протокол), запрос уходит в правильном протоколе автоматически (catalog-driven), без ручной настройки — иначе tool-calls деградируют.

### Не-регрессия встроенных (критично)

8. Для **каждого** встроенного провайдера (anthropic, openAI, gemini, агрегаторы, локальные) форма исходящего запроса, заголовки, формат инструментов, reasoning-параметры и лимиты токенов **не меняются ни на байт**. Добавление, изменение или удаление `.vibe/providers.json` никогда не влияет на то, как строится запрос встроенного провайдера.

9. При отсутствии `.vibe/providers.json` (или когда в нём нет активных динамических провайдеров) поведение системы идентично состоянию до этой фичи.

### Reasoning / «думанье»

10. Модель с включённым `reasoning` **думает**: размышления отображаются в чате как блок размышления, и у провайдера запрашивается reasoning (уровень усилия учитывается).

11. `reasoning.effort` задаёт допустимые уровни усилия; для thinking-моделей при незаданном уровне применяется разумный **высокий** дефолт (модели вроде K2.7 Code рассчитаны думать по умолчанию).

12. `reasoning: false` (или отсутствие reasoning у не-reasoning модели) → размышление не запрашивается, обычная генерация.

13. **Round-trip размышления:** там, где провайдер этого ожидает, прошлые размышления модели сохраняются между ходами, чтобы модель не теряла цепочку и не зацикливалась на повторных вызовах.

14. **`extraBody` уходит в тело запроса дословно** (для квирков провайдера/модели, например `thinking: { type: "enabled" }`), без переинтерпретации со стороны VibeIDE. `extraBody` — аддитивен: он не затирает поля, которыми управляет VibeIDE и которые сам не называет.

15. **Никакого тихо-пустого ответа у reasoning-моделей:** лимит выходных токенов всегда отправляется, чтобы размышление не съело весь бюджет и не осталось пустого контента (ответ 200 без текста).

### Надёжность под Moonshot / openai-compatible (часть — to validate)

16. Схемы инструментов, отправляемые динамическому/openai-compatible провайдеру, нормализуются так, чтобы провайдер их принял: вложенным property-схемам без `type` проставляется `type`; локальные `$ref`/`$defs` инлайнятся. Нормализация **идемпотентна** (повтор = один раз) и сохраняет комбинаторы (`anyOf`/`oneOf`/`allOf`).
    - **To validate:** прежде чем включать собственную нормализацию, подтвердить, что именно не покрывает текущий AI-SDK путь, и применять только там, где нужно.

17. Нормализация схемы применяется **только** на динамическом/openai-compatible пути; схемы инструментов встроенных провайдеров не трогаются (следствие инварианта 8).

18. Идентификаторы вызовов инструментов (`tool_call_id`), которыми обмениваемся с провайдером, приводятся к провайдеро-безопасной форме (≤64 символов, санитизация) **симметрично**: id в вызове ассистента и id в соответствующем результате инструмента остаются равны после нормализации — связь «вызов ↔ результат» не рвётся.
    - **To validate:** определять провайдеров, которым это нужно, по конфигу/каталогу, либо применять всегда на динамическом пути — уточнить по факту, что делает AI-SDK слой сам.

19. Поведения из (16)–(18) включаются только для провайдеров, которым они нужны; провайдер, которому это не требуется, не видит изменений.

### Edge cases (сводно)

20. Динамик без активных статических моделей → в выборе модели не появляется (нечего отправлять), это не ошибка.
21. MCP-инструмент с property только из `enum` и без `type` → после нормализации (16) принимается провайдером, а не отклоняется.
22. Очень длинный `tool_call_id` от провайдера → усечён/санитизирован по (18), парность сохранена.
23. Несколько активных динамических провайдеров одновременно → каждый маршрутизируется и отправляет независимо; их порядок в выборе — по `order` (существующее поведение), на отправку не влияет.

## Open questions

- **Open question (gating Phase C):** нормализацию схемы и ограничение `tool_call_id` включать всегда на динамическом пути или только при детекте конкретного провайдера (по baseURL/каталогу)? Решается замером того, что AI-SDK путь уже делает сам (инварианты 16, 18 помечены «to validate»).
