Приоритетный стек `.vibe/`, детерминированный enforcement constraints, CortexIDE как стартовая точка.

---

## [договорённость] Приоритетный стек настроек `.vibe/`

**Контекст:** риск #32, зафиксирован в docs/v1/config/overview.md.

**Суть:** иерархия (от высшего к низшему): Enterprise locked → Global constraints → Profile constraints → Directory rules → Mode constraints. Mode НЕ может снять ограничения уровней 1-3.

**Применение:** при реализации профилей и enterprise policy.

---

## [архитектура] `.vibe/constraints.json` — детерминированный enforcement, не промпт

**Контекст:** риск #43 из idea.md, зафиксирован в docs/v1/config/constraints.md.

**Суть:** constraints.json должен форсироваться через **детерминированную sandbox-прослойку до агента** — IDE физически блокирует вызов при нарушении constraint, независимо от инструкций в промпте. Если это только промпт-инструкция — LLM может её проигнорировать.

**Применение:** при реализации constraints в Фазе 1 — не ограничиваться промптом.

---

## [архитектура] CortexIDE как стартовая точка, не постоянная зависимость

**Контекст:** риск #50 из idea.md.

**Суть:** CortexIDE — 87 звёзд, один активный автор. `FORK_CHANGES.md` документирует все отклонения чтобы можно было перейти на прямой форк `microsoft/vscode` при необходимости. Мониторинг: нет коммитов 60+ дней → алерт в CI.

**Применение:** не строить глубокие зависимости от CortexIDE-специфичных API.
