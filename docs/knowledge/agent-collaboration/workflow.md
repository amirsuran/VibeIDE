# Agent Workflow Rules

Поведенческие правила взаимодействия агента с автором проекта. Дополняют формальный протокол `CLAUDE.md` («план → подтверждение → код») практическими исключениями и расширениями.

---

## Меньше mid-task confirmations [workflow] [foot-gun]

**Контекст.** `CLAUDE.md` формализует протокол «план → ожидать подтверждения → код» для каждой задачи. После двух подтверждений «да» подряд на тривиальные follow-up-ы пользователь явно сказал «Сделай всё без лишних вопросов».

**Суть.** Внутри **уже одобренной** задачи не требовать дополнительных подтверждений на очевидные follow-up-ы (локализация хардкод-строк после фикса, коммит после прохождения `compile-check-ts-native`, smoke-fix после ошибки сборки, дополнение nls-ключей). Главный план задачи всё равно проходит через «Стоп. Подтверди?» — это правило не отменяется.

**Применение.**
- Признаки «можно без подтверждения»: те же файлы, то же намерение, та же логика; пользователь уже сказал «да» родителю.
- Снова просить подтверждение если: реальная развилка из нескольких равно правомерных подходов, изменение скоупа, рискованная необратимая операция (push/force/destructive).
- См. [batch autonomous execution](#batch-autonomous-execution-on-explicit-ask) — расширение этого правила на явное cross-task batch-разрешение.

## Batch autonomous execution on explicit ask [workflow]

**Контекст.** `CLAUDE.md` мандат «plan → confirm → code» по умолчанию. Правило «mid-task confirmation» выше покрывает только follow-up-ы внутри ОДНОЙ задачи. Иногда пользователь явно разрешает прогнать сразу серию задач из ранее показанного списка кандидатов.

**Суть.** При явной формулировке «Выполни оставшиеся пункты все без вопросов», «do remaining items without asking», «автоном» — пропускать per-item plan→confirm и прогонять подряд то, что уже было показано как кандидаты в недавнем `AskUserQuestion` или в proposal.

**Применение.**
- Триггер только на ЯВНУЮ авторизационную формулировку — не на «продолжай» / «ещё пункт».
- Скоуп = только что перечисленные кандидаты, не безграничное «делай что считаешь нужным».
- После каждого пункта показывать Diff Report (формат CLAUDE.md) — пользователь должен видеть что прилетело.
- Останавливаться на TS / test failures и докладывать — не пробивать поломку.
- В конце batch'а — единая summary всех завершённых пунктов + спросить про commit.

**Пример.** В конце roadmap-grind сессии с ~25 коммитами в очереди после фразы «Выполни оставшиеся пункты все без вопросов» прогнаны i18n brand allowlist + audit hook + commands.audit registration в одном turn'е, с одной summary в конце вместо трёх plan→approve циклов.

## Логирование model stalls [workflow] [reference]

**Контекст.** Пользователь настроил журнал инцидентов про обрывы/зависания LLM-ассистента 2026-05-12 как способ накопить статистику и выйти на корневую причину.

**Суть.** Когда пользователь жалуется на обрыв/зависание/остановку модели — открывать [`docs/knowledge/chat-ux/model-stalls.md`](../chat-ux/model-stalls.md) и фиксировать инцидент по шаблону, заданному в самом файле.

**Применение.**
- Триггеры: «модель повисла / оборвалась / остановилась / замолчала / зависла», «обрыв», «обрыв связи», «перестала думать», «не вызвала тул», «псевдо-tool-call», EN: `stall / stuck / cut off / truncated`. Полный список с алиасами — внутри `model-stalls.md`.
- Действия: прочитать файл → добавить запись СВЕРХУ журнала с инкрементным номером → если набирается ≥3 похожих — выделить в раздел «Паттерны».
- Проверить, нет ли уже похожего инцидента, и сослаться на него — не плодить дубликаты.
- Сам `model-stalls.md` декларирует workflow, шаблон записи, триггерные слова, черновые гипотезы — он источник истины.

## Сборки — только по явному запросу [workflow] [foot-gun]

**Контекст.** Пользователь сам контролирует когда делать локальный билд (`local-windows.ps1`) и релиз (`release-windows.ps1`). Любая «инициативная» сборка агентом — это лишнее время, шум в git/tags и риск выкатить недоготовое в прод. Правило поставлено 2026-05-18 после серии правок UI чата, где достаточно было `compile + run-dev` для проверки.

**Суть.** После любой правки в `src/` агент делает **только** `npm run compile && .\run-dev.bat` (TS-сборка + запуск dev-инстанса), чтобы пользователь мог проверить изменения. Никаких `scripts\release-windows.ps1`, `scripts\local-windows.ps1`, `npm run gulp …` и прочих билдов — пока пользователь явно не написал `собери` / `билд` / `build` / «делай локальный билд» / «выкатывай в прод».

**Применение.**
- После Edit/Write в `src/` → запустить `npm run compile && .\run-dev.bat` (или `npm run compile-check-ts-native` для быстрой проверки TS, если правка ещё не финальна).
- НЕ запускать `local-windows.ps1` / `release-windows.ps1` — пользователь сам выберет: «делай локальный билд» или «выкатывай в прод с релиза».
- Это правило **переопределяет** предыдущую секцию «Не отправлять тестировать без подтверждённой пересборки» — теперь dev-инстанс через `run-dev.bat` заменяет `local-windows.ps1` как стандартный способ проверки правок.
- Триггер на сборку — только явная команда от пользователя; даже «проверь что собирается» = `compile-check-ts-native`, не полный билд.

## Не отправлять тестировать без подтверждённой пересборки [workflow] [foot-gun]

**Контекст.** Пользователь тестирует правки переносом `D:\Projects\VibeCode\VibeIDE-win32-x64\*` поверх `D:\Progs\AI\VibeIDE\` (свежий бинарник, data-папка нетронута). Если внести правку в `src/` и попросить тестировать без пересборки, пользователь увидит результат **прошлого** билда — те же ошибки, реакция «такое ощущение, что ты ничего не делал». Инцидент 2026-05-17 при работе над утечкой listener'ов в чате (правка про globalThis-кеш TTP отправлена тестировать без `local-windows.ps1`).

**Суть.** После любого изменения в `src/` обязательно: (1) запустить `.\scripts\local-windows.ps1` (с подходящими флагами `-SkipReact` / `-SkipCompile` для скорости) → (2) дождаться exit 0 → (3) только потом просить пользователя перенести файлы и тестировать.

**Применение.**
- Правка только в `*.html`/ассетах workbench → `-SkipReact -SkipCompile` (минут 2).
- Правка в `src/vs/workbench/contrib/vibeide/browser/react/src/**` → без `-SkipReact` (нужно ребандлить tsup).
- Правка в обычном `src/` без react → без `-SkipCompile` (нужен TS-compile).
- Перед запуском билда — `npm run compile-check-ts-native` (быстрый TS-check, чтобы не тратить 6 минут билда на правке с typo).
- В сообщении пользователю — явно перечислить какие файлы скопировать (он сам не обязан помнить путь `VibeIDE-win32-x64`).
