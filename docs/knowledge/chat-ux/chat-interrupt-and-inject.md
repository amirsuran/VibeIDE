# Прерывание турна и подмешивание контекста

← [Knowledge Index](../README.md) · связано: [modes-and-policies.md](modes-and-policies.md)

Что происходит при abort посреди хода агента и как добавить контекст к следующему хопу, не прерывая выполнение.

---

## [баг] HTTP 400 «duplicate tool_call id» после прерывания на середине tool-call

**Контекст:** пользователь прерывает модель посреди хода (чтобы дополнить контекст), затем отправляет сообщение — провайдер отвечает `HTTP 400: invalid params, duplicate tool_call id: call_...` (2026-06).

**Суть:** сборка сообщений для OpenAI-протокола — `prepareMessages_openai_tools` в `browser/convertToLLMMessageService.ts` — на каждый tool-result дописывает `tool_call` в `tool_calls[]` ближайшего assistant-сообщения и пушит `role:'tool'`. **Дедупликации по `id` не было.** Прерывание на середине tool-call оставляет в истории недо-завершённый/повторно сыгранный `tool_call` (abort-путь в `chatThreadService` фиксирует частичный ход + маркер `interrupted_streaming_tool`); на следующем запросе тот же `id` попадает в `tool_calls[]` дважды (или дублируется tool-result) → провайдер отклоняет весь запрос. Лечилось только тем, что пользователь руками останавливал процесс и слал отдельное сообщение про ошибку.

**Применение:** в `prepareMessages_openai_tools` заведён `seenToolCallIds = new Set<string>()`; повторный `tool_call id` → `continue` (пропускаем всю пару tool_call + tool-result) + `vibeLog.warn`. Санитайзинг на границе отправки корректен независимо от апстрим-причины: провайдер требует уникальные `tool_call id` в одном assistant-сообщении. **Дальше (durable):** не плодить дубль в истории на самом abort-пути (`chatThreadService`), чтобы Set был лишь страховкой.

## [архитектура] «Подмешать контекст к следующему хопу» (inject без прерывания)

**Контекст:** во время многохопного хода агента пользователь хочет добавить данные, но сейчас единственный путь — прервать (abort) и отправить новое сообщение (что и провоцирует баг выше). Нужна возможность поставить контекст в очередь, чтобы агент подхватил его на следующем хопе сам.

**Суть (дизайн):**
- **Данные:** per-thread буфер `pendingInjection` в `ChatThreadState` (рядом со `stagingSelections`) — текст (+ опц. selections/images).
- **Точка инъекции:** многохоповый цикл в `chatThreadService` (вверху каждой итерации, где читаются `messages` и до `prepareLLMChatMessages`). Если буфер не пуст — добавить его как **реальное user-сообщение** в позиции «после tool-результатов текущего хопа, перед следующим LLM-вызовом» (валидная позиция; не ломает пары tool_call↔result), пометить флагом (видно в транскрипте «добавлено в ходе выполнения»), очистить буфер.
- **UX (реализовано):** пока агент работает, **Enter** в инпуте кладёт текст в очередь (`addPendingInjection`) + уведомление «подмешается на следующем хопе (N)»; раньше Enter при выполнении был no-op. Escape остаётся abort. Основной Enter при простое не меняется. Файл — `react/src/sidebar-tsx/SidebarChat.tsx` (`onInject` + ветка в keydown).
- **Края:** если ход завершился с непустым буфером — остаток не теряется: дренаж стоит вверху КАЖДОЙ итерации, поэтому на следующем запуске он подмешается (как доп. user-сообщение). Авто-отправка остатка отдельным турном сразу по завершении — опциональное улучшение.

**Применение:** реализовано (2026-06-18). Метод `IChatThreadService.addPendingInjection(threadId, text)`; дренаж `_drainPendingInjections` в цикле `_runChatAgent` (перед чтением `chatMessages`). Опционально: чип-счётчик у инпута. До этой фичи обходной путь был abort + сообщение (с фиксом дедупа `tool_call id` 400 больше не возникает).

## [баг/правило] Слабая модель завершает ход ТЕКСТОМ вместо `vibe_complete` (цикл автопилота)

**Контекст:** модель (MiniMax/deepseek через агрегатор) пишет «Готово / задача выполнена / шагов не осталось» текстом вместо вызова `vibe_complete` — автопилот шлёт корректирующий нудж по кругу, ход не закрывается (2026-06).

**Суть:** машинерия уже есть (`chatThreadService` ~6634): детект «ход текстом без тула» → нудж + `forceToolUse` (`tool_choice:'required'`, `vibeide.agent.forceToolUseOnNudge`, дефолт on) + implicit-complete по исчерпании бюджета. **Но `tool_choice:'required'` работает только в native-tool режиме.** Слабые модели обычно в **XML-fallback** (нет `specialToolFormat` → native-тулзы и `tool_choice` не шлются) → форсить нечем, а текст нуджа «вызови `vibe_complete`» не сопоставляется у такой модели с эмитом XML-тега → снова проза → цикл.

**Применение:** нудж сделан **mode-aware** — если у модели `!caps.specialToolFormat` (XML-режим), к текстовому корректирующему нуджу добавляется точная форма для вывода:
```
<vibe_complete>
<summary>что сделано, 1–3 предложения</summary>
</vibe_complete>
```
Только для текстового кейса (не question/empty). `let corrective` + `xmlCompleteHint` в `chatThreadService` (~6654). Не трогаем «авто-вызов за модель» — форк осознанно этого не делает. Совет «усилить system-prompt текстом» (от самой модели) — поверхностный: контракт уже в описании `vibe_complete`; реальный пробел был именно XML-форма для слабых моделей.
