## [архитектура] A. Cache-friendly сборка промпта (prompt caching)

**Статус:** ✅ внедрено 2026-06-07 (фаза 1: repo_context + implicit skills → user turn, лог cached tokens).
✅ Фаза 2 (2026-06-07): `cache_control` для Anthropic-протокола — system уезжает внутрь messages
с брейкпоинтом + брейкпоинт на последнем сообщении (инкрементальный кеш всей истории на
следующий ход). Anthropic без явных брейкпоинтов не кеширует НИЧЕГО: наблюдали 23k input/ход
→ упор в org-лимит 30k TPM за 11 итераций. Прокси (Zen), срезающий поле, безвреден — оно аддитивно.
Смежно: автоожидание минутного rate-limit (`vibeide.chat.rateLimitAutoWait*`).
✅ РАЗГАДАНО (2026-06-07, после дампа `vibeide.debug.dumpFullPrompt` с tool-блоками): историю
переписывал ЛЕГАСИ-кламп `safetyTrim` в `prepareMessages` — захардкоженный `MAX_INPUT_TOKENS_SAFETY
= 20_000` («под OpenAI 30k TPM»): всё сверх 20k input-токенов МОЛЧА резалось до 120-символьных
огрызков, начиная с «самых тяжёлых» сообщений (старые крупные tool-result'ы). Отсюда: смерть кеша
с ~11-й итерации (префикс мутировал каждый ход), циклы перечитывания файлов (от прочитанного
оставалось 120 символов), скачки `in:` 23k↔39k. Фикс: порог вынесен в `vibeide.chat.maxInputTokensSafety`,
дефолт 0 = ВЫКЛЮЧЕН (payload ограничен окном модели; TPM — автопаузой), срабатывание логируется.
Урок: NO SILENT TRIMS — любой механизм, переписывающий историю, обязан логировать каждый факт.

**Контекст:** OpenAI-совместимые роуты (openCode Zen и др.) автоматически кешируют префикс
запроса (>1024 токенов): совпавшая байт-в-байт голова промпта тарифицируется со скидкой до 90%.
Наш `convertToLLMMessageService` системно ломал префикс: `<repo_context>` (выдача RepoIndexer,
меняется каждый ход) и implicit-skill-хинты (зависят от текста последнего сообщения) дописывались
В КОНЕЦ SYSTEM-сообщения — т.е. в самое начало запроса. Каждый ход — полный холодный input
(в логах прогона: 12.5k→13.4k input на каждом ходу без единого кеш-хита).

**Суть:** правило — «стабильное в начало, динамика в конец». System (инструкции + tool-схемы +
discovery скиллов) должен быть байт-стабилен между ходами; всё пер-ходовое (retrieval, хинты,
языковая директива) — в ПОСЛЕДНИЙ user-ход (механизм уже существовал для `/skill:`-тел —
model-stalls.md #002). Step A.5-стабы детерминированы (один раз стаб — навсегда тот же текст),
поэтому компакция кеш-дружелюбна: каждый ход инвалидирует только хвост от свежего стаба.
Budget-fill `<chat_summary>` дописывается в конец system — голова system остаётся кешируемой.

**Применение:** при любом новом инжекте контекста в промпт спрашивать: «это меняется от хода
к ходу?» Если да — только в последний user-ход, НИКОГДА в system. Эффект мерить по
`cachedInputTokens` в логе TokenBudget (видно после фазы 1).

**Антипаттерны:** инжект per-turn данных в system «потому что это контекст»; таймстампы/random
ID в начале промпта; переписывание уже отправленных сообщений истории нестабильным текстом.

## [инструмент] B. Конденсер вывода терминала (RTK-подобный)

**Статус:** ✅ внедрено 2026-06-07 (`common/terminalOutputCondenser.ts`).

**Контекст:** вывод `run_command` клампился только по размеру (хвост/голова), не по смыслу.
Тест-раннеры (pytest/jest/cargo), пакетные менеджеры и сборщики генерируют сотни строк
прогресс-шума («ok», прогресс-бары, дубли) — модель платит за их чтение input-токенами.
Идея RTK (github.com/rtk-ai/rtk): сжимать вывод ДО попадания в контекст.

**Суть:** чистая функция `condenseTerminalOutput()` в `common/`: дедуп подряд идущих
одинаковых/прогресс-строк, счётчики вместо повторов, сохранение окон вокруг ошибок
(error/fail/exception/traceback), итоговых строк раннеров («N passed, M failed») и
головы/хвоста. Срабатывает только на длинном выводе (порог), конфиг
`vibeide.terminal.condenseOutput` (default on).

**Применение:** конденсер ставится ПЕРЕД размерным клампом в terminalToolService — смысловое
сжатие первично, грубое обрезание остаётся страховкой.

**Антипаттерны:** агрессивно резать вывод НЕизвестного формата (потеря сигнала хуже расхода);
сжимать короткий вывод (модель теряет точность цитирования без выигрыша).

## [архитектура] C. Дешёвая модель для служебных вызовов (model routing)

**Статус:** ✅ внедрено 2026-06-07 (`vibeide.chat.auxiliaryModel`).

**Контекст:** служебные LLM-вызовы (генерация плана execution-режима, авто-заголовки тредов,
внутренние summaries) шли на основную — обычно дорогую — модель пользователя. Статья: делить
работу на «мозг» (планирование, ревью) и «руки» (классификация, выжимки, заголовки).

**Суть:** настройка `vibeide.chat.auxiliaryModel` (строка `provider/model`, default `''` = выкл,
текущее поведение). Когда задана и валидна — служебные вызовы маршрутизируются на неё; основной
агентский цикл НЕ трогается. Невалидное значение — молчаливый fallback на основную модель
(служебный вызов не должен падать из-за опечатки в настройке).

**Применение:** новые служебные вызовы (не-агентские, не-чатовые) обязаны идти через
резолвер auxiliary-модели, а не напрямую через model selection фичи Chat.

**Связано:** [llm-and-context.md](../architecture/llm-and-context.md) — провайдеры и каталог.

## [архитектура/правило] D. Бюджеты субагентов отделены от session-бюджета (single chokepoint)

**Статус:** ✅ 2026-07-11 (`next`).

**Контекст:** `vibeTokenBudgetService` = per-session safety-гейт основного агента (2M токенов, гейт `vibeide.safety.sessionTokenLimit`). Субагенты (curated роли VA + внутренние explore/implement-step) исполняются настоящим LLM↔tools циклом и имеют СВОЙ бюджет (`vibeide.subagent.maxTokens`). Но их сенды шли через тот же `sendLLMMessageService.sendLLMMessage` → `checkBudget()` + `recordUsage()`, поэтому спенд роли (а) капал в session-счётчик (пользователь: «неправильно»), (б) мог заблокировать роль сессионным гейтом.

**Суть:** `sendLLMMessage` — единственная точка, где трогается session-бюджет (`checkBudget` перед отправкой, `recordUsage` в `onFinalMessage`). Флаг `excludeFromSessionBudget` в `ServiceSendLLMMessageParams` (renderer-only, деструктурируется ВОН из `proxyParams` → не уходит по IPC) отключает обе операции. Все LLM-вызовы субагентов идут через ОДИН чокпоинт — `browser/vibeSubagentRunnerService.ts` (`this._llm.sendLLMMessage({... excludeFromSessionBudget: true})`), поэтому одного флага в одном месте достаточно.

**Применение:** session-бюджет = ТОЛЬКО прямые ходы основного агента. Любой новый «фоновый»/изолированный LLM-цикл со своим бюджетом (субагенты, будущие headless-раннеры) обязан ставить `excludeFromSessionBudget: true`, иначе его расход задваивается в session-гейте и может ложно заблокировать основную сессию. Пер-роль учёт токенов/$ живёт в отчёте маршрута (`subagentCostEstimate`), не в session-бюджете.

**Гоча отображения:** индикатор `TokenBudgetInline` (`SidebarHistory.tsx`) показывал `43 332·4%` — не битьё, а `ru-RU` пробел-разделитель тысяч («43 332») вплотную к `·%`. Форма `X (P%)` локаль-независима; не возвращать `·`-склейку между числом с разделителем и процентом.

## [архитектура/правило] E. Контекст-окно субагента отделено от main-метра (тот же singleton)

**Статус:** ✅ 2026-07-11 (`next`). Продолжение [[D]] — вторая течь после session-бюджета.

**Контекст:** `IVibeContextGuardService` (индикатор «Контекст/Окно») — **общий singleton**, который кормит `convertToLLMMessageService.prepareLLMChatMessages` через `updateUsage`/`setTruncationStats`/`setCalibrationFactor` (5 сайтов). Субагент-раннер строит промпт той же `prepareLLMChatMessages` → контекст роли перетирал метр main-треда (на смоуке «Окно» скакало до 6%, пока роль читала файлы). Session-бюджет к этому моменту уже был отвязан (§D), но контекст-метр — отдельный канал, его не закрывало.

**Суть:** флаг `skipContextGuardUpdate` в опциях `prepareLLMChatMessages`. Внутри — `const ctxGuard = skipContextGuardUpdate ? undefined : this.contextGuardService`, все 5 мутаций через `ctxGuard?.` (чтения `getStatus` не трогаем — они в других функциях). Раннер шлёт `skipContextGuardUpdate: true`. `recordActualPromptTokens` (само-калибровка) субагент не зовёт вовсе — она только в main-цикле (`chatThreadService`), так что калибровка тоже чистая.

**Применение:** любой изолированный LLM-цикл со своим контекстом (субагенты, будущие headless-раннеры) обязан ставить `skipContextGuardUpdate: true` — иначе его промпт перетирает индикатор контекста пользовательского треда. Два независимых канала утечки субагента в main: (1) session-бюджет → флаг `excludeFromSessionBudget` в `sendLLMMessage` (§D); (2) контекст-метр → флаг `skipContextGuardUpdate` в `prepareLLMChatMessages` (§E). Оба — на своих чокпоинтах.

**Отображение контекста роли:** НЕ отдельный статус-бар-метр (YAGNI), а инлайн в живом индикаторе «🧩 Роль «X» работает… (~48k / 400k)». Телеметрия: `SubagentRunRequest.onProgress` (per-hop из раннера) → `entry.liveTokensUsed`/`tokenQuota` + `onStatusChanged` → хук `useSubagentActivity`.
