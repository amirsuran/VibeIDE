# Бенчмарк дисциплины минимализма (vibeide.global.minimalismMode)

Методика измерения эффекта «лестницы минимализма» на код, который пишет агент.
Подход: A/B-прогоны одинаковых задач с выключенной и включённой дисциплиной,
скоринг по git diff. Автоматизацию headless-сессий сознательно не строим —
прогоны запускаются вручную, скрипт только считает метрики.

## Протокол

1. **Репозиторий-полигон.** Один и тот же проект-шаблон для всех прогонов
   (любой средний full-stack шаблон; главное — одинаковое начальное состояние:
   фиксированный коммит, чистое дерево).
2. **Набор задач.** 8–12 реалистичных фич-тикетов (не багфиксы — на них
   минимализм почти не влияет). Формулировки фиксируются заранее и не меняются
   между режимами.
3. **Условия.** Для каждой задачи минимум по 2 прогона на условие:
   - `A` — `vibeide.global.minimalismMode: "off"`;
   - `B` — `"full"` (и опционально `C` — `"ultra"`).
   Одна и та же модель, один провайдер, один режим чата (agent), автопилот включён.
4. **Прогон.** Новый чат-тред → текст тикета → дождаться завершения → **не**
   править руками. После прогона:
   ```sh
   node scripts/vibe-bench-diff-score.mjs --label "task03-B-full" >> bench-results.jsonl
   git reset --hard <базовый-коммит> && git clean -fd   # вернуть полигон в исходное
   ```
5. **Сравнение.** Медианы по условию: added/removed/net LOC, файлов затронуто.
   Плюс ручная проверка «фича работает» (запуск/тест) — метрика без этого
   бессмысленна: −50% кода при сломанной фиче не результат.

## Метрики (`vibe-bench-diff-score.mjs`)

| Поле | Что означает |
|---|---|
| `filesChanged` | затронуто файлов |
| `insertions` / `deletions` | строки по `git diff --shortstat` |
| `netLines` | insertions − deletions |
| `newFiles` | создано новых файлов |
| `label` | метка прогона (`--label`) |

## Правила честности

- Прогон, где агент не довёл фичу (упал, зациклился) — **бракуется в обеих
  ветках**, а не только в неудобной.
- Не смешивать модели/провайдеры между условиями одной задачи.
- Разговорный вывод агента в метрики не входит — только git diff.
- Результаты — в jsonl, по строке на прогон; сводку считать любым способом
  (jq/скрипт/таблица).
