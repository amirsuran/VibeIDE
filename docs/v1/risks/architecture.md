# Риски — архитектура

## #8 — Автообновление + migration path

**При обновлении меняется схема `auditLogService.ts`, `.vibe/context.md` — без migration path пользователь теряет данные.**

**Решение:** `check for updates` через GitHub Releases API.  
Для каждого релиза с изменением схемы — явный migration script + тест на upgrade с реальными данными.

---

## #9 — Electron CVE + зависимости

**Форки наследуют устаревшие версии Electron с известными уязвимостями.**

**Решение:** Два CI-джоба: Electron CVE tracking + npm audit на lockfile при каждом PR.

---

## #15 — Атомарность inline diff + rollback

**Частичное применение чанков + git stash pop = конфликты слияния внутри механизма отката.**

**Решение:** Явная атомарность: либо всё, либо ничего, либо явный промпт.  
Тест на частичное применение diff.

> Зафиксировать в Фазе 0.

---

## #19 — Архитектурный конфликт: снапшоты vs git

**`rollbackSnapshotService.ts` и `gitAutoStashService.ts` могут конфликтовать.**

**Решение:**
- Агент работает только с `refs/vibe/checkpoint-*`, никогда не трогает `stash@{0}`
- Явная проверка detached HEAD + fallback во временную директорию
- Явная проверка submodules + отдельный fallback

---

## #20 — Vector store без внешних зависимостей

**`vectorStore.ts` использует Qdrant/Chroma — внешние сервисы. Противоречит «работает из коробки».**

**Решение:** sqlite-vec или LanceDB как дефолт. Qdrant/Chroma — опциональный backend.

> ⚠️ sqlite-vss deprecated с 2024 года, использовать sqlite-vec.

---

## #21 — treeSitterService.ts — монорепо

**Полный парсинг 500k+ файлов заморозит IDE на минуты. При неготовности индекса — молчаливая деградация.**

**Решение:** Инкрементальный индекс + лимиты + прогресс-бар + явный fallback «индекс не готов».

---

## #22 — auditLogService.ts — блокировка event loop

**Синхронная запись на каждый AI-action фризит UI именно тогда когда агент активен.**

**Решение:** Запись асинхронная и буферизованная. Ротация логов (дефолт 30 дней).

---

## #25 — RAG-индексирование vs privacy gate

**`repoIndexerService.ts` + `vectorStore.ts` читают весь репо. В privacy-режиме при подключении к сети — утечка кода.**

**Решение:** В privacy-режиме — никаких сетевых запросов при индексировании.  
Embedding-модель = принудительно локальная в privacy-режиме.

---

## #37 — Rollback race condition в multi-agent режиме

**Два агента создают checkpoint одновременно — один перезапишет другой.**

**Решение:** Явный mutex/lock при создании checkpoint. Порядок фиксируется по wall clock + agent ID.  
**Спроектировать в Фазе 0, реализовать при multi-agent (Фаза 3b).**

---

## #43 — `.vibe/constraints.json` — детерминизм

**Если constraints — только промпт-инструкция, LLM может её проигнорировать.**

**Решение:** Детерминированная sandbox-прослойка до агента: блокировка вызова независимо от агента.

> Зафиксировать в Фазе 0, реализовать в Фазе 1.

---

## #45 — Agent action history — scope и persistence

**«Текущая сессия» нигде не определена — история может теряться при crash.**

**Решение:** История персистируется в `auditLogService.ts` (не только в памяти). Привязана к `sessionId`.  
Retention = общий retention аудит-лога.

---

## #50 — CortexIDE dormancy

**CortexIDE — 87 stars, один автор. При уходе — нет upstream.**

**Решение:** `FORK_CHANGES.md` документирует все отклонения для переноса на чистый VS Code форк.  
Мониторинг: нет коммитов 60+ дней → алерт в CI → обсуждение перехода на прямой форк.

---

## #51 — `.vibe/` format versioning

**Без версионирования схемы обновление IDE молча сломает конфиги пользователей.**

**Решение:** Поле `"vibeVersion"` в каждом `.vibe/` файле. Migration script при смене схемы.  
JSON Schema публикуется на GitHub Pages.

---

## #53 — Hot-reload `.vibe/` во время активного агента

**Редактирование `.vibe/constraints.json` пока агент активен = inconsistent state.**

**Решение:** Изменения вступают в силу только при следующем tool-call или явном Reload.  
Banner при редактировании во время активного агента.

---

## #54 — Multi-root workspace + `.vibe/` иерархия

**Workspace isolation, иерархия `.vibe/` и Smart context picker не определены для multi-root.**

**Решение:**
- Каждый корень = независимая `.vibe/` конфигурация
- Global constraints применяются ко всем корням
- Smart context picker индексирует все корни, уважает per-root `.vibe/ignore`

> Зафиксировать в Фазе 0.

---

## #64 — `.vibe/context.md` race condition с VSCodeSyncFiles

**Агент обновляет `context.md` + VSCodeSyncFiles синхронизирует = конфликт на двух устройствах.**

**Решение:** Агент пишет через атомарный write с `sessionId`-меткой.  
При конфликте побеждает запись с более новым `agentTimestamp`.

---

## #72 — Context limit mid-task

**Агент начинает задачу, контекст заполняется → агент молча теряет начало задачи.**

**Решение:** При 90% лимита — диалог: compact / продолжить с риском / отменить + снапшот.  
Live-индикатор заполнения во время выполнения.

---

## #78 — `models.json` community PR model не масштабируется

**Модели выходят каждые 1-2 недели. PR review + merge + релиз = 2 недели задержки.**

**Решение:** `models.json` хостится на CDN endpoint (`registry.vibeide.io/models.json`).  
IDE делает GET с ETag кешированием. Community PRs — в отдельный manifest-репо.

---

## #89 — Checkpoint storage растёт без ограничений

**`refs/vibe/checkpoint-*` — не трогается `git gc`, нет TTL. Несколько ГБ за месяц.**

**Решение:** Дефолт: последние 50 + именованные. Автопрунинг включён.  
`vibe doctor --full` предупреждает при >1GB.
